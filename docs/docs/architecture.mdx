# Architecture

## Monorepo Layout

Bundt uses a Turborepo-managed Bun workspace with four top-level directories:

```
apps/        Deployable CLIs and servers (published with bin entries)
packages/    Publishable npm libraries (consumed as dependencies)
internal/    Private shared packages (never published to npm)
crates/      Rust crates via Cargo workspace
```

## Package Relationships

```
  @bundt/cleo          @bundt/dxdocs        @bundt/loclaude
  (CLI app)            (CLI app)            (CLI app)
       |                    |                    |
       v                    v                    v
  internal/shared      internal/shared      internal/shared
                            |
                            v
                       @bundt/waavy (planned SSR for docs)

  @bundt/hateoas       @bundt/waavy         @bundt/signals
  (library)            (library)            (planned)
       |                    |
       v                    v
     react             react, react-dom
```

All CLI apps share `cac` for argument parsing and `picocolors` for terminal output. Build utilities are shared via `@bundt/internal-build-utils`.

## Build System

### Turborepo

`turbo.json` defines the task graph. Each package declares `build`, `typecheck`, `test`, and `clean` scripts. Turborepo resolves dependencies, parallelizes independent tasks, and caches outputs.

### Bun.build

CLI apps use `Bun.build()` directly instead of a bundler config file. Each app has a `build/` directory:

```
build/
  base.ts      Shared config (external deps, common options)
  bun_.ts      Bun-target build
  node_.ts     Node-target build (optional)
```

The `concurrent_build()` utility runs multiple build targets in parallel.

### Library Packages

Libraries like `@bundt/hateoas` use source-level exports — `package.json` exports point directly to TypeScript source files. No build step is needed during development.

Libraries like `@bundt/waavy` compile to `out/` with type declarations generated by `tsc`.

## Conventions

### TypeScript

- **Strict mode** everywhere: `noImplicitAny`, `noUncheckedIndexedAccess`, `noImplicitOverride`
- **`verbatimModuleSyntax`** — use `import type` for type-only imports
- **No `any`** — use `unknown` with type guards
- **No `enum`** — use `as const` objects or union types
- **No default exports** — named exports only (config files excepted)
- **No `.js` extensions** in TypeScript imports
- **`jsx: react-jsx`** — automatic JSX runtime, no React import needed

### CLI Pattern

All CLIs use `cac` for argument parsing with a consistent structure:

```ts
import cac from 'cac';

const cli = cac('tool-name');

cli.command('action', 'Description').action(async () => {
  // implementation
});

cli.help();
cli.version('0.1.0');
cli.parse();
```

### Dependencies

- **Bun** as runtime, package manager, bundler, test runner
- **Turborepo** for workspace orchestration
- **Zod 4** for validation schemas
- **cac** for CLI argument parsing
- **picocolors** for terminal colors
