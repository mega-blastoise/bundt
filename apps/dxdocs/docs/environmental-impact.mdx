# Environmental Impact

The internet accounts for roughly [2-4% of global carbon emissions](https://www.bbc.com/future/article/20200305-why-your-internet-habits-are-not-as-clean-as-you-think) — on par with the airline industry. Every byte transferred costs energy: at the data center, across the network, and on the device rendering the page.

Documentation sites are read millions of times. The architecture choices behind them matter more than most people think.

dxdocs takes the most aggressive approach possible: **ship nothing**. Zero client-side JavaScript. Pure static HTML and CSS. Every page is pre-rendered at build time and served as a plain document.

## Why This Matters

Most documentation platforms ship a JavaScript framework to the browser. That JS has to be downloaded, parsed, compiled, and executed on every single page load — before the reader can even see the content they came for.

This isn't a theoretical problem. The median web page in 2025 transfers over 2,200 KB according to the [HTTP Archive](https://httparchive.org/reports/state-of-the-web). A large chunk of that is JavaScript that the user never asked for.

dxdocs pages are typically **under 30 KB total** — HTML, CSS, and syntax-highlighted code included. No framework runtime. No hydration. No bundle.

## How dxdocs Compares to Mintlify

[Mintlify](https://mintlify.com) is a popular hosted documentation platform. It looks great. But under the hood, it ships a full Next.js application to every reader. That means every page load includes React, a client-side router, hydration code, and telemetry scripts — even though the content is static text.

Here's what that looks like in practice:

| | dxdocs | Mintlify |
|---|---|---|
| JavaScript shipped to reader | **0 KB** | ~200-400 KB |
| HTML rendering | Pre-rendered at build time | Server-rendered, then hydrated on client |
| Framework in browser | None | React + Next.js |
| Hosting model | Any static host (CDN) | Mintlify's servers |
| Network requests per page | 1-2 (HTML + CSS) | 15-30+ (JS chunks, API calls, telemetry) |
| Works with JS disabled | Yes | Partially |

### What this means for the environment

Every unnecessary kilobyte has a cost. When a Mintlify docs page loads ~300 KB of JavaScript that dxdocs simply doesn't need, that difference multiplies across every reader, every page view, every day.

For a documentation site with 100,000 monthly page views:

- **Mintlify**: ~30 GB/month of JavaScript transfer alone
- **dxdocs**: ~0 GB/month of JavaScript transfer (because there is none)

That's 30 GB of data that doesn't need to be generated by servers, transmitted across networks, downloaded by devices, or parsed by CPUs. Less data means less energy at every point in the chain.

### It's not just bandwidth

JavaScript is the most expensive type of content a browser handles. A 300 KB JS bundle costs more energy than a 300 KB image because the browser has to parse, compile, and execute the code — work that heats CPUs, drains batteries, and is especially punishing on older or lower-powered devices.

By shipping zero JS, dxdocs doesn't just save bandwidth. It reduces CPU time on the reader's device, which directly translates to less energy consumed and longer battery life on phones and laptops.

## Same Visual Quality, No Waste

The point of dxdocs isn't to be a stripped-down alternative. It has the same features you'd expect from a premium docs platform:

- Sidebar navigation with nested groups
- Auto-generated table of contents with scroll tracking
- Syntax highlighting with Shiki (11 languages, proper tokenized colors)
- Light and dark theme support
- Callouts, cards, steps, and other rich components
- Responsive layout with mobile breakpoints

The difference is that all of this is rendered to static HTML at build time. The browser receives a finished document. There's nothing to hydrate, no components to mount, no router to initialize. The page is ready the instant it arrives.

## Hosting Efficiently

Because dxdocs produces plain HTML files, you can serve them from any CDN or static host — Cloudflare Pages, Netlify, Vercel, GitHub Pages, S3, or even a simple nginx server. Static files are the most cache-friendly, most CDN-friendly, most energy-efficient content the web can serve.

There's no application server running, no serverless functions spinning up, no database queries executing. Just files on disk, served over HTTP.

<Callout variant="tip">
Set long cache headers on your built output. Since dxdocs pages are static, browsers and CDNs can cache them aggressively, reducing repeat transfer to near zero.
</Callout>

## The Simplest Architecture Wins

There's a common assumption that achieving a polished, modern-looking documentation site requires a client-side framework. Mintlify, Docusaurus, Nextra — they all ship React to the browser.

dxdocs proves that assumption wrong. You write MDX, you run a build, and you get HTML. The result looks the same. The reader experience is the same (or better, since pages load instantly). The only difference is what you *don't* ship.

The greenest code is code that doesn't run.
